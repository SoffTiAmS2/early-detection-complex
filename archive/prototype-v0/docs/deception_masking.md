# Deception masking

## Как это работает в реальных deception-системах

Реальная deception-платформа не просто открывает порт. Она создает правдоподобную легенду узла:

- имя хоста похоже на рабочий сервер;
- открытые сервисы соответствуют роли;
- баннеры не конфликтуют друг с другом;
- логи и ответы выглядят ожидаемо;
- приманка расположена в правильном сетевом сегменте;
- обращение к приманке почти всегда считается подозрительным сигналом.

Пример: если узел называется `office-filesrv-01`, то логично показывать SSH, SMB/FTP или HTTP-службу администрирования. Если узел называется `mine-telemetry-gw`, логичнее показывать HTTP-панель и промышленный протокол, а не случайный набор портов.

## Как это сделано здесь

Файл `config/project.json` хранит выбранную легенду:

- `honeypots[]` - выбранные honeypot на сенсоре;
- `honeypots[].services` - какие сервисы открыть внутри конкретного honeypot;
- `honeypots[].settings` - настройки конкретного honeypot;
- `mask.hostname` - имя хоста-легенды;
- `mask.os` - ОС-легенда;
- `mask.department` - контекст размещения;
- `mask.asset_tag` - инвентарная метка.

Эти параметры задаются в `config/project.json`. После этого `center/orchestrator/generate.py` создает ignored-артефакты:

- `.env`;
- `docker-compose.yml`;
- `cowrie/etc/cowrie.cfg`;
- `cowrie/etc/userdb.txt`;
- `cowrie/honeyfs/...`;
- `README.md` для каждого сенсора.

## Что реально меняется

Единый образ `edc-sensor` собирается из `sensor/Dockerfile`: Cowrie ставится из исходников в Python venv, чтобы сборка работала на ARM-платах. Генератор создает `cowrie.cfg`, где задаются:

- fake hostname;
- SSH version banner;
- backend;
- auth policy;
- download limit;
- включение SFTP;
- SSH/Telnet listen endpoints внутри контейнера.

Host-порты задаются в generated `docker-compose.yml` на основе `config/project.json`, поэтому можно выбрать, на каком порту сенсор будет принимать SSH/Telnet honeypot-трафик.

Генератор также создает Cowrie-персону:

- `userdb.txt` с управляемым фейковым логином и паролем;
- `/etc/hostname`, `/etc/issue.net`, `/etc/motd`, `/etc/passwd`, `/etc/group` внутри `honeyfs`;
- домашнюю директорию пользователя, `.bash_history`, заметки и приманочные файлы;
- директорию `downloads` для артефактов атакующего.

При запуске контейнер собирает `fs.pickle` из `honeyfs` через Cowrie `createfs` и указывает его в `[shell] filesystem`. Это означает, что выбор профиля влияет не только на документацию, но и на поведение сенсора внутри SSH/Telnet-сессии.

## Ограничение текущей реализации

Сейчас реально подключен Cowrie. Другие honeypot нужно добавлять только тогда, когда для них будет готов настоящий контейнер, конфиг и сбор логов, а не просто пункт в интерфейсе.
